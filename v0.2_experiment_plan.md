# v0.2 Experiment Plan

Status: next run plan. Labels should be frozen before running v0.2 results.

## Question

Do semantic retrieval methods change the failure-class distribution compared with lexical retrieval?

The key outcome is not only higher retrieval accuracy. The important question is whether a method changes the safety direction of errors:

- fewer false-certainty errors,
- fewer downgrade misses,
- fewer overblocks,
- more benign misses when retrieval is imperfect.

## Fixed Inputs

Use the same current inputs for the first v0.2 run:

- same 21 memory objects,
- same 10 internal scenarios,
- same expected memory labels,
- same expected action labels,
- same action severity order.

Do not change scenario labels after seeing semantic retrieval results.

## Baselines Already Present

- `tfidf_content_only`
- `tfidf_metadata_content`
- `tfidf_keyword_expanded`
- `bm25_content_only`
- `bm25_metadata_content`
- `bm25_keyword_expanded`

## Candidate Additions

Add at least one embedding baseline:

- local embedding model if available,
- API embedding model if cost and privacy are acceptable,
- hybrid BM25 + embedding score if the single embedding baseline is stable.

Optional additions:

- top-k recall,
- mean reciprocal rank,
- top-k policy aggregation,
- latency and cost per scenario.

## Required Output Table

Every run should report:

| Strategy | Top-1 retrieval | Top-k recall | Action correct | False certainty | Downgrade miss | Overblocking | Benign miss | Mean latency |
|---|---:|---:|---:|---:|---:|---:|---:|---:|

## Claim Test

The v0.2 run is useful if it can show one of these:

- a semantic method fixes `s02_overclaim_eval_results`;
- a semantic method has the same top-1 accuracy as lexical retrieval but a different failure-class distribution;
- a semantic method improves retrieval but creates a worse action-class failure, such as false certainty;
- lexical retrieval and semantic retrieval both fail the same way, suggesting the issue is policy aggregation or scenario labeling rather than representation.

## Stop Condition

Do not write a stronger public claim unless the run produces a clear comparison table and the limitations remain attached to the result.
